# torch.compile and CUDA Optimizations: Making the GPU Go Fast

Training a transformer is compute-bound. The model itself — attention, matrix
multiplies, normalization — is just math. The question is how fast you can
execute that math. This doc explains every optimization cola-coder uses to
squeeze maximum throughput out of your RTX 3080/4080, what each one actually
does at the hardware level, and how to debug them when something breaks.

If you have ever profiled a Node.js server and found that switching from
`JSON.parse` to a streaming parser doubled throughput, you already understand
the principle. Same work, fewer wasted cycles. That is what this entire
document is about.

## 1. The Performance Stack

When you write `loss = model.compute_loss(input_ids)` in Python, here is what
actually happens:

```
Your Python code          ←  high-level, interpreted, slow
    ↓
PyTorch framework         ←  dispatches operations to C++ / CUDA
    ↓
CUDA kernels              ←  compiled GPU programs (matrix multiply, softmax, etc.)
    ↓
GPU hardware              ←  tensor cores, SRAM, HBM (the actual silicon)
```

**TS analogy:** This is exactly like the web performance stack:

```
Your TypeScript code      ←  high-level, readable
    ↓
V8 JavaScript engine      ←  interprets → baseline compiles → optimizes hot paths
    ↓
Machine code              ←  x86/ARM instructions generated by TurboFan JIT
    ↓
CPU hardware              ←  branch predictors, caches, SIMD units
```

In both stacks, the bottleneck is rarely the bottom layer (hardware is fast).
The bottleneck is in the translation layers — how efficiently your high-level
code becomes low-level instructions. Every optimization in this document targets
one of those translation layers:

| Optimization | Layer it targets | What it eliminates |
|---|---|---|
| torch.compile | Python → CUDA | Python interpreter overhead, redundant memory reads |
| TF32 | CUDA → Hardware | Uses tensor cores for fp32 matmuls (7x throughput) |
| Mixed precision | CUDA → Hardware | Halves memory bandwidth, doubles tensor core throughput |
| Flash Attention | Algorithm → CUDA | O(n²) memory → O(n) by tiling into SRAM |
| Fused AdamW | Python → CUDA | 1 kernel launch instead of ~8 for optimizer step |
| cuDNN benchmark | CUDA → Hardware | Auto-selects fastest convolution algorithm |
| Non-blocking transfers | CPU ↔ GPU | Overlaps data transfer with computation |

## 2. torch.compile: JIT Compilation for GPU Code

### What it does

Without `torch.compile`, every line of your model's `forward()` method runs
through the Python interpreter. Each PyTorch operation — every `matmul`, every
`softmax`, every `+` — is a separate call from Python into C++/CUDA. The GPU
executes the operation, returns control to Python, Python moves to the next
line, and the cycle repeats.

This is like running JavaScript with every function call going through
`eval()` — it works, but you are paying interpreter overhead on every single
operation.

`torch.compile` traces your model's forward pass, captures the full computation
graph (every operation and how tensors flow between them), and compiles it into
optimized GPU code. The Python interpreter is removed from the hot path entirely.

```python
# In trainer.py, line ~107:
self.model = torch.compile(self.model, mode="default")
```

After this call, `self.model` is a compiled wrapper. The first time you call
`model(input_ids)`, PyTorch traces the computation, compiles it (this takes
30-60 seconds), and caches the compiled code. Every subsequent call runs the
compiled version directly.

### The V8 TurboFan analogy

This is exactly what V8's TurboFan JIT compiler does for JavaScript:

```
JavaScript (no JIT):
  for (let i = 0; i < 1000000; i++) {
    result += arr[i] * scale;   // interpreted: ~100ns per iteration
  }

JavaScript (TurboFan JIT):
  // V8 detects this is a hot loop, compiles to native x86:
  // vmulps ymm0, ymm1, ymm2    // AVX2 SIMD: ~1ns per iteration
```

```
PyTorch (no compile):
  x = self.norm(x)              # Python → C++ → CUDA kernel launch → wait
  x = self.attention(x)         # Python → C++ → CUDA kernel launch → wait
  x = self.ffn(x)               # Python → C++ → CUDA kernel launch → wait
  # 3 kernel launches, 3 round-trips through Python

PyTorch (torch.compile):
  # All three ops fused into one or two kernels
  # No Python in the loop
  # Intermediate tensors stay in GPU registers/SRAM instead of global memory
```

### What the compiler actually optimizes

1. **Operator fusion**: Combines multiple operations into a single GPU kernel.
   Instead of writing `norm(x)` result to GPU memory, then reading it back
   for `attention(x)`, the compiled code keeps intermediate values in fast
   on-chip memory (like keeping values in CPU registers instead of writing
   to RAM after every operation).

2. **Eliminates Python overhead**: No interpreter in the loop. No Python object
   creation for intermediate tensors. No GIL contention.

3. **Memory planning**: The compiler knows the entire computation graph upfront,
   so it can reuse tensor memory instead of allocating new buffers.

4. **Kernel selection**: Picks optimal CUDA kernel implementations for your
   specific tensor shapes (e.g., choosing a tiled matmul for 768×768 vs a
   different strategy for 4096×4096).

### Compilation modes

torch.compile has three modes:

| Mode | What it does | Memory overhead | Speedup |
|---|---|---|---|
| `"default"` | Operator fusion + Python elimination | None | 20-40% |
| `"reduce-overhead"` | Above + CUDA graphs (pre-records kernel launches) | High (pre-allocates all GPU memory) | 30-50% |
| `"max-autotune"` | Above + benchmarks hundreds of kernel variants | Very high at compile time | 35-55% |

**We use `"default"`** because `"reduce-overhead"` uses CUDA graphs, which
pre-allocate GPU memory for the entire computation. On a 16GB card (RTX 4080)
training a 350M+ model, this can push you into OOM territory. The 20-40%
speedup from `"default"` is already substantial, and it does not cost extra VRAM.

```python
# trainer.py — why "default" mode:
# "default" mode: good speedup without extra VRAM overhead.
# "reduce-overhead" uses CUDA graphs which pre-allocate memory
# and can cause OOM on 16GB cards.
self.model = torch.compile(self.model, mode="default")
```

TS analogy: `"default"` is like V8's TurboFan with normal optimizations.
`"reduce-overhead"` is like pre-compiling your entire app with AOT and pinning
it in memory — faster if you have the RAM, but you might run out.

### The `_orig_mod.` prefix problem

When you call `torch.compile(model)`, PyTorch wraps the original model inside
a compiled container. The original model becomes an attribute called `_orig_mod`.
This means every key in the state dict gets prefixed:

```
Before compile:  model.state_dict() keys:
  "tok_emb.weight"
  "layers.0.attention.q_proj.weight"
  "layers.0.attention.k_proj.weight"
  ...

After compile:  model.state_dict() keys:
  "_orig_mod.tok_emb.weight"
  "_orig_mod.layers.0.attention.q_proj.weight"
  "_orig_mod.layers.0.attention.k_proj.weight"
  ...
```

This is a problem for checkpoints. If you save with the prefix, the checkpoint
only loads into a compiled model. If you later want to load the checkpoint for
inference without torch.compile (e.g., on CPU, or on a different machine), the
keys won't match.

**Our solution** (in `checkpoint.py`): Strip the prefix on save, add it back
on load.

```python
# checkpoint.py — save_checkpoint():
# torch.compile wraps keys with "_orig_mod." prefix, so we strip that too
# to keep checkpoint format consistent regardless of compilation.
raw_state = model.state_dict()
state_dict = {}
for k, v in raw_state.items():
    clean_key = k.removeprefix("_orig_mod.")
    # ... (also skip tied weights)
    state_dict[clean_key] = v.contiguous()

# checkpoint.py — load_checkpoint():
# Handle torch.compile: if model is compiled, keys need _orig_mod. prefix
# Checkpoints always store clean keys (no prefix) for portability.
if hasattr(model, "_orig_mod"):
    state_dict = {f"_orig_mod.{k}": v for k, v in state_dict.items()}
```

TS analogy: This is like how bundlers (webpack, esbuild) might wrap your module
exports in a runtime container. If you serialize state, you need to unwrap it
first so the serialized format is independent of the bundler. Our checkpoints
are "unbundled" — they work whether the model is compiled or not.

### First-step compilation warmup

The first training step after calling `torch.compile` is much slower (30-60
seconds) because PyTorch is tracing and compiling the computation graph. This is
a one-time cost per session. The trainer prints a warning about this:

```
torch.compile: ON (first few steps will be slower due to compilation)
```

This is exactly like V8's warmup phase: the first few thousand calls to a
function run through the interpreter, then TurboFan kicks in and subsequent
calls are 10-100x faster. Same principle, different scale.

## 3. TF32: Free Performance on Ampere+ GPUs

### What TF32 is

TF32 (TensorFloat-32) is a number format introduced with NVIDIA's Ampere
architecture (RTX 30xx series). It is a 19-bit floating point format that
uses:
- 1 sign bit (same as fp32)
- 8 exponent bits (same as fp32 — same range)
- 10 mantissa bits (same as fp16 — less precision than fp32's 23 bits)

```
fp32:  [1 sign] [8 exponent] [23 mantissa]  = 32 bits
TF32:  [1 sign] [8 exponent] [10 mantissa]  = 19 bits
fp16:  [1 sign] [5 exponent] [10 mantissa]  = 16 bits
bf16:  [1 sign] [8 exponent]  [7 mantissa]  = 16 bits
```

The key insight: TF32 has the **same range as fp32** (8 exponent bits) but
**less precision** (10 mantissa bits vs 23). For matrix multiplications during
neural network training, this reduced precision has negligible impact on final
model quality.

### Why it is free performance

When you enable TF32, the GPU's tensor cores perform matrix multiplications
using TF32 internally, even though your tensors are stored as fp32. The inputs
are truncated from 23 to 10 mantissa bits, multiplied using tensor cores at
much higher throughput, and the result is stored back as fp32.

This is transparent — your code does not change, your tensors stay fp32, but
matrix multiplies run up to **8x faster** on tensor cores vs pure fp32 CUDA
cores.

```python
# trainer.py — _setup_cuda_optimizations():
# TF32: uses Tensor Cores for float32 matmuls at ~7x the throughput
# of pure fp32, with slightly less precision (fine for training).
torch.backends.cuda.matmul.allow_tf32 = True
torch.backends.cudnn.allow_tf32 = True
```

Two flags because there are two separate code paths:
- `cuda.matmul.allow_tf32`: Affects `torch.matmul`, `@` operator, linear layers
- `cudnn.allow_tf32`: Affects cuDNN operations (convolutions, which some
  architectures use)

### Precision impact

The mantissa truncation (23 bits → 10 bits) means TF32 has about 3 decimal
digits of precision instead of 7. For training, this is fine because:

1. Gradient updates are inherently noisy (stochastic gradient descent)
2. The optimizer's momentum buffers smooth out per-step noise
3. Empirically, models trained with TF32 reach the same final loss as pure fp32

There are no known cases where TF32 causes training divergence or measurable
quality loss for transformer training. NVIDIA enables it by default in PyTorch
2.0+ for exactly this reason.

TS analogy: Imagine if JavaScript's `Number` type had a "fast math mode" that
rounded to 3 decimal places internally but gave you 8x faster arithmetic. For
most web app calculations (prices, coordinates, percentages), you would never
notice the difference. TF32 is that, but for matrix math.

### Hardware support

| GPU | Architecture | TF32 Support |
|---|---|---|
| RTX 3080 | Ampere (GA102) | Yes |
| RTX 4080 | Ada Lovelace (AD103) | Yes |
| RTX 20xx | Turing | No (ignored silently) |
| Any CPU | — | No (ignored silently) |

The flags are safe to set unconditionally. On hardware without TF32 tensor
cores, PyTorch silently ignores them.

## 4. Mixed Precision: bf16, fp16, and torch.autocast

### The core idea

Neural network training has two kinds of numbers:

1. **Model weights and activations** — the tensors flowing through the network.
   These can tolerate reduced precision because small rounding errors average
   out over millions of parameters.

2. **Optimizer state** — the momentum buffers and variance estimates in AdamW.
   These accumulate small gradient updates over thousands of steps. Even tiny
   rounding errors compound, so they must stay in fp32.

Mixed precision means: run the model in half precision (16-bit), but keep the
optimizer in full precision (32-bit). You get the speed and memory benefits of
16-bit for the expensive forward/backward pass, and the numerical stability of
32-bit for the optimizer.

TS analogy: This is like using `Float32Array` for bulk image processing
(where speed matters and small errors are invisible) but `Number` (fp64) for
your financial calculations (where every cent matters). Different precision for
different jobs.

### bf16 vs fp16: the range vs precision tradeoff

```
fp32:  [1 sign] [8 exponent] [23 mantissa]   range: ±3.4 × 10³⁸
bf16:  [1 sign] [8 exponent]  [7 mantissa]   range: ±3.4 × 10³⁸  (same!)
fp16:  [1 sign] [5 exponent] [10 mantissa]   range: ±65,504       (much smaller)
```

**bf16 (bfloat16):**
- Same range as fp32 (8 exponent bits). Gradients can be tiny (1e-30) or large
  (1e30) and bf16 handles both without overflow or underflow.
- Less precision (7 mantissa bits ≈ 2 decimal digits). But for neural network
  training, this is fine.
- **No GradScaler needed.** Since gradients never overflow/underflow, you can
  just cast and go.
- Available on Ampere+ (RTX 30xx/40xx).

**fp16 (float16):**
- Smaller range (5 exponent bits). Gradients smaller than ~6e-8 underflow to
  zero. Gradients larger than 65,504 overflow to infinity.
- More precision than bf16 (10 mantissa bits ≈ 3 decimal digits).
- **Requires GradScaler.** Because small gradients underflow, you multiply the
  loss by a large scale factor before backward, then divide gradients back
  down after. The scaler dynamically adjusts this factor to avoid overflow.
- Works on all CUDA GPUs (including Turing / RTX 20xx).

### Which to use

| GPU | Precision | GradScaler | Config |
|---|---|---|---|
| RTX 4080 (Ada) | bf16 | Not needed | `precision: "bf16"` |
| RTX 3080 (Ampere) | bf16 | Not needed | `precision: "bf16"` (also works) |
| RTX 3080 (Ampere) | fp16 | Required | `precision: "fp16"` |
| RTX 20xx (Turing) | fp16 | Required | `precision: "fp16"` |

Both the RTX 3080 and 4080 support bf16. The 3080 has full bf16 tensor core
support on Ampere. Use bf16 unless you have a specific reason to prefer fp16
(slightly more mantissa precision in exchange for needing GradScaler).

### How it works in our code

```python
# trainer.py — __init__():
self.use_bf16 = config.training.precision == "bf16"
self.use_fp16 = config.training.precision == "fp16"
self.scaler = GradScaler("cuda", enabled=self.use_fp16)  # only active for fp16
```

```python
# trainer.py — training loop:
# Pre-computed outside the loop for speed:
use_amp = self.use_bf16 or self.use_fp16
amp_dtype = torch.bfloat16 if self.use_bf16 else torch.float16

# Inside the loop:
with autocast(device_type=self.device, dtype=amp_dtype, enabled=use_amp):
    loss = self.model.compute_loss(input_ids)
    scaled_loss = loss * inv_accum

# Backward pass (scaler is a no-op when disabled, i.e., for bf16)
self.scaler.scale(scaled_loss).backward()

# ... later:
self.scaler.unscale_(self.optimizer)       # Unscale gradients for clipping
torch.nn.utils.clip_grad_norm_(...)        # Clip in fp32
self.scaler.step(self.optimizer)           # Step (skips if gradients have inf/nan)
self.scaler.update()                       # Adjust scale factor
```

### What torch.autocast actually does

`torch.autocast` is a context manager that intercepts PyTorch operations and
automatically casts their inputs to half precision where it is safe to do so.
Not all operations are cast — PyTorch maintains an internal allowlist:

| Operations cast to fp16/bf16 | Operations kept in fp32 |
|---|---|
| Linear layers (matmul) | Loss functions (cross-entropy) |
| Convolutions | Softmax |
| Batch/layer normalization | Log/exp operations |
| Attention (Q×K, attn×V) | Reduction operations (sum, mean) |
| Activation functions | Small matrix operations |

This is why it is called "mixed" precision — different operations run at
different precisions within the same forward pass, automatically.

TS analogy: Think of `autocast` like TypeScript's type narrowing. Inside the
context manager, PyTorch "narrows" tensor types to fp16/bf16 where it knows
it is safe, and keeps them at fp32 where precision matters. You do not have to
manually cast each tensor — the framework handles it.

### Memory savings

Half precision halves the memory for model weights and activations:

| Component | fp32 | bf16/fp16 | Savings |
|---|---|---|---|
| 125M model weights | 500 MB | 250 MB | 250 MB |
| Activations (batch=8, seq=1024) | ~2 GB | ~1 GB | ~1 GB |
| Optimizer state | 2 GB (always fp32) | 2 GB (always fp32) | 0 |
| **Total** | **~4.5 GB** | **~3.25 GB** | **~1.25 GB** |

The optimizer state stays in fp32 regardless. This is why the optimizer often
dominates VRAM usage for large models — AdamW keeps two fp32 buffers (momentum
and variance) per parameter, so optimizer state is 2x the model size in fp32.

### GradScaler deep dive (fp16 only)

The problem with fp16 is underflow. During backpropagation, gradients can be
very small (1e-8 or smaller). In fp16, the smallest representable positive
number is ~6e-8. Anything smaller becomes zero, and zero gradients mean the
model stops learning.

GradScaler fixes this by:

1. **Before backward**: Multiply the loss by a large scale factor (starts at
   65536, adjusts dynamically).
2. **During backward**: All gradients are proportionally larger, so they don't
   underflow.
3. **Before optimizer step**: Divide gradients by the scale factor to get the
   true values.
4. **After step**: Check if any gradients were inf/nan. If so, skip this step
   and reduce the scale factor. If not, consider increasing it.

```
Without scaler (fp16):
  loss = 0.5
  gradient of some weight = 1e-9   → rounds to 0 in fp16 → weight never updates

With scaler (fp16):
  scaled_loss = 0.5 * 65536 = 32768
  gradient of some weight = 1e-9 * 65536 = 6.5e-5   → representable in fp16!
  after unscaling: 6.5e-5 / 65536 = 1e-9             → correct gradient value
```

With bf16, gradients of 1e-9 are representable directly (8 exponent bits give
range down to ~1e-38), so no scaler is needed. This is why bf16 is strictly
simpler to use.

## 5. Flash Attention via F.scaled_dot_product_attention

### The O(n²) memory problem

Standard attention computes:

```
Attention(Q, K, V) = softmax(Q × Kᵀ / √d) × V
```

The intermediate matrix `Q × Kᵀ` has shape `(seq_len, seq_len)`. For a
sequence length of 1024, that is a 1024×1024 = 1,048,576 element matrix.
**Per head, per batch element.** With 12 heads and batch size 8:

```
Attention matrix memory = batch × heads × seq_len × seq_len × bytes_per_element
                        = 8 × 12 × 1024 × 1024 × 2 (bf16)
                        = 192 MB   ← just for the attention scores!
```

At sequence length 2048, this quadruples to ~768 MB. At 4096, it is ~3 GB.
This is the O(n²) memory scaling that makes long sequences expensive.

### How Flash Attention works

Flash Attention (Dao et al., 2022) never materializes the full attention
matrix. Instead, it tiles the computation into blocks that fit in the GPU's
SRAM (shared memory / L1 cache), which is much faster than global memory (HBM)
but much smaller (~200 KB per streaming multiprocessor vs 10-16 GB of HBM).

The algorithm:

1. Split Q, K, V into blocks that fit in SRAM
2. For each block of Q, iterate over blocks of K and V
3. Compute partial attention scores in SRAM (fast)
4. Accumulate the output without ever writing the full attention matrix to HBM (slow)
5. Use an online softmax algorithm to combine partial results correctly

```
Standard attention:
  GPU SRAM (fast, small) → [load Q block] → [compute Q×K block] → write full QK to HBM
  GPU HBM  (slow, large) ← [full QK matrix sits here, 192 MB]
  GPU SRAM (fast, small) → [load QK block] → [softmax] → write back to HBM
  GPU HBM  (slow, large) ← [softmax result sits here, 192 MB]
  GPU SRAM (fast, small) → [load softmax block] → [× V block] → write output to HBM

Flash attention:
  GPU SRAM (fast, small) → [load Q,K,V blocks] → [compute attention] → [accumulate output]
  GPU HBM  (slow, large) ← [only the final output, no intermediate matrices]
```

TS analogy: This is like the difference between `array.map().filter().reduce()`
(creates intermediate arrays in heap memory) vs a single `for` loop that does
all three operations in one pass (keeps everything in stack/registers). Same
result, way less memory allocation.

### How we use it

```python
# attention.py — forward():
# Use PyTorch's fused scaled_dot_product_attention (SDPA).
# Auto-dispatches to Flash Attention 2 > memory-efficient > math fallback.
# is_causal=True is faster than constructing + applying an explicit mask.
if use_cache:
    # Inference with KV-cache: q_len != kv_len, need explicit mask
    output = F.scaled_dot_product_attention(
        q, k, v, attn_mask=mask, dropout_p=drop_p, scale=self.scale,
    )
else:
    # Training: is_causal=True — no mask tensor needed
    output = F.scaled_dot_product_attention(
        q, k, v, dropout_p=drop_p, is_causal=True, scale=self.scale,
    )
```

We also explicitly enable the Flash and memory-efficient backends in
`_setup_cuda_optimizations()`:

```python
# trainer.py:
torch.backends.cuda.enable_flash_sdp(True)
torch.backends.cuda.enable_mem_efficient_sdp(True)
```

PyTorch's `F.scaled_dot_product_attention` (SDPA) is a dispatch function that
automatically selects the best available backend:

1. **Flash Attention 2** — fastest, requires head_dim ≤ 256 and certain dtypes
2. **Memory-efficient attention** (xFormers) — slightly slower, wider support
3. **Math fallback** — standard implementation, used as last resort

You don't choose which backend runs — PyTorch picks the best one for your
tensor shapes, dtypes, and GPU.

### The `is_causal=True` optimization

During training, we need causal masking — token at position `i` can only
attend to positions `0..i`, not future tokens. The naive way is to create an
explicit mask tensor:

```python
# Naive: allocate a (seq_len × seq_len) mask tensor
mask = torch.triu(torch.ones(seq_len, seq_len), diagonal=1).bool()
mask = mask.float().masked_fill(mask, float('-inf'))
output = F.scaled_dot_product_attention(q, k, v, attn_mask=mask)
```

This wastes memory (another O(n²) tensor) and is slower because the kernel
has to load and apply the mask from global memory. With `is_causal=True`,
the kernel applies the triangular mask internally using simple index
comparisons — no mask tensor needed:

```python
# Efficient: kernel applies causal mask via index math, no tensor allocated
output = F.scaled_dot_product_attention(q, k, v, is_causal=True)
```

During inference with KV-cache, `q_len != kv_len` (we're generating one token
at a time against the full cached context), so `is_causal=True` can't be used —
we pass an explicit mask instead. But during training, `is_causal=True` is
strictly better.

### Performance impact

| Metric | Standard attention | Flash Attention | Improvement |
|---|---|---|---|
| Memory (seq_len=1024, batch=8) | ~200 MB for attention | ~0 MB intermediate | ~200 MB saved |
| Memory (seq_len=2048, batch=8) | ~800 MB for attention | ~0 MB intermediate | ~800 MB saved |
| Wall-clock time | Baseline | 2-3x faster | Saves 50-70% of attention time |
| Scaling | O(n²) memory | O(n) memory | Enables longer sequences |

Flash Attention is the single biggest optimization for transformer training.
Without it, training at sequence length 2048 with batch size 8 would require
significantly more VRAM just for attention intermediates, potentially forcing
you to reduce batch size and slow down training.

## 6. Fused AdamW Optimizer

### What "fused" means

A standard AdamW optimizer step involves multiple separate operations per
parameter:

```python
# Pseudocode for one AdamW parameter update (non-fused):
# Each line = one CUDA kernel launch = one Python→C++→CUDA round trip

grad = param.grad                          # read gradient
m = beta1 * m + (1 - beta1) * grad        # kernel 1: update momentum
v = beta2 * v + (1 - beta2) * grad ** 2   # kernel 2: update variance
m_hat = m / (1 - beta1 ** t)              # kernel 3: bias correction
v_hat = v / (1 - beta2 ** t)              # kernel 4: bias correction
param -= lr * (m_hat / (sqrt(v_hat) + eps) + weight_decay * param)
                                           # kernel 5-8: div, sqrt, add, multiply, subtract
```

Each kernel launch has overhead: ~5-10 microseconds for the Python→CUDA
dispatch, plus the kernel can't start until the previous one finishes (unless
the compiler can overlap them). For a model with hundreds of parameter tensors,
this adds up.

The **fused** implementation combines all of these into a single CUDA kernel
per parameter tensor:

```python
# Fused AdamW: ONE kernel does everything
# Read grad, m, v, param → compute update → write m, v, param
# No intermediate reads/writes to global memory
```

```python
# optimizer.py:
return AdamW(
    param_groups,
    lr=learning_rate,
    betas=betas,
    eps=1e-8,
    fused=torch.cuda.is_available(),  # Use faster fused kernel on GPU
)
```

### Performance impact

The fused optimizer typically saves 10-20% of the optimizer step time. For a
125M model, the optimizer step is ~10% of total step time, so this translates
to roughly a 1-2% overall speedup. Small, but free.

TS analogy: This is like the difference between:

```typescript
// Slow: multiple passes over the array
const doubled = arr.map(x => x * 2);
const filtered = doubled.filter(x => x > 10);
const sum = filtered.reduce((a, b) => a + b, 0);

// Fast: single pass
let sum = 0;
for (const x of arr) {
  const d = x * 2;
  if (d > 10) sum += d;
}
```

Same result, but the single-pass version avoids allocating intermediate arrays
and iterating multiple times. Fused AdamW is the single-pass version of the
optimizer step.

## 7. cuDNN Benchmark Mode

### What it does

cuDNN (CUDA Deep Neural Network library) provides multiple implementations
of operations like convolutions, each optimized for different input shapes.
When `benchmark=True`, cuDNN runs each implementation on your actual input
tensors, measures wall-clock time, and caches the fastest one:

```python
# trainer.py — _setup_cuda_optimizations():
torch.backends.cudnn.benchmark = True
```

### Why the first batch is slow

On the first batch, cuDNN runs a benchmark for every unique input shape it
encounters. This can add several seconds to the first step. After that, the
optimal algorithms are cached and every subsequent step uses them directly.

```
Step 0:  2.3 seconds  ← cuDNN benchmarking (+ torch.compile tracing)
Step 1:  0.8 seconds  ← torch.compile still compiling some kernels
Step 2:  0.12 seconds ← everything warm, this is your actual throughput
Step 3:  0.12 seconds
...
Step N:  0.12 seconds
```

### When to disable it

cuDNN benchmark assumes input shapes are constant across batches. If your
sequence lengths vary (e.g., variable-length batching without padding),
cuDNN re-benchmarks on every new shape, which is slower than not benchmarking
at all.

In cola-coder, we use fixed sequence lengths (all sequences padded/truncated to
`max_seq_len`), so benchmark mode is always beneficial.

Note: While transformers themselves don't use convolutions, some operations
in the pipeline may go through cuDNN paths, and the benchmark flag also signals
to PyTorch to optimize other operation dispatching. The flag is cheap to enable
and never harmful when input shapes are fixed.

## 8. Non-Blocking GPU Transfers

### The problem

Moving data from CPU RAM to GPU memory (HBM) goes over the PCIe bus. On a
PCIe 4.0 x16 connection (RTX 3080/4080), theoretical bandwidth is ~32 GB/s,
but practical throughput is ~12-15 GB/s due to overhead.

A training batch might be 4-8 MB of token IDs. At 12 GB/s, that is 0.3-0.7 ms
of transfer time. By default, this transfer blocks — the GPU sits idle while
data copies over.

### Non-blocking transfers

With `non_blocking=True`, the transfer is queued on a separate CUDA stream
and returns immediately. The GPU can continue executing the current computation
while the next batch's data is transferring in the background:

```python
# trainer.py — training loop:
# non_blocking=True overlaps CPU→GPU transfer with compute
input_ids = batch["input_ids"].to(self.device, non_blocking=True)
```

```
Without non_blocking:
  [GPU idle: copying batch 2] [GPU compute: forward/backward batch 2] [GPU idle: copying batch 3] ...

With non_blocking:
  [GPU compute: forward/backward batch 1] [GPU compute: forward/backward batch 2] ...
  [PCIe: copying batch 2                ] [PCIe: copying batch 3                ] ...
    ↑ happens in parallel on a different hardware unit
```

TS analogy: This is exactly like `fetch()` being non-blocking in JavaScript.
Instead of waiting for the network response before doing anything, you start
the fetch and keep running other code. The data arrives in the background.

```typescript
// Blocking (like default .to()):
const data = await fetch(url);  // CPU sits idle waiting
processData(data);

// Non-blocking (like .to(non_blocking=True)):
const dataPromise = fetch(url);  // Returns immediately
doOtherWork();                   // Keep working while network transfers
const data = await dataPromise;  // Data is already here
```

### pin_memory in DataLoader

For non-blocking transfers to work efficiently, the CPU memory should be
"pinned" (page-locked). Normal CPU memory can be swapped to disk by the OS;
pinned memory is guaranteed to stay in physical RAM, which allows the GPU's
DMA engine to copy directly without going through the CPU.

Our DataLoader uses `pin_memory=True` by default when CUDA is available (this
is PyTorch's default for CUDA dataloaders). The data loading pipeline in
`dataset.py` produces tensors in pinned CPU memory, enabling efficient
non-blocking transfers.

## 9. Combined Impact: The Full Stack in Action

Here is an estimate of how each optimization contributes to a training step
for our 125M (small) model on an RTX 4080 with bf16:

| Optimization | Effect | Approximate impact |
|---|---|---|
| No optimizations (fp32, no compile) | Baseline | ~18 tok/s |
| + TF32 | Tensor core matmuls | ~35 tok/s (+95%) |
| + bf16 mixed precision | Half memory bandwidth | ~55 tok/s (+57%) |
| + Flash Attention (SDPA) | O(n) attention memory | ~70 tok/s (+27%) |
| + torch.compile | Fused kernels, no Python overhead | ~86 tok/s (+23%) |
| + Fused AdamW | Single-kernel optimizer | ~88 tok/s (+2%) |
| + Non-blocking transfers | Overlapped data movement | ~90 tok/s (+2%) |

These numbers are approximate and vary by model size, sequence length, and
batch size. The key takeaway: **the combined effect is roughly 5x faster than
naive fp32 training.** A training run that would take 10 days takes 2 days
instead.

The optimizations are not simply additive — they interact. Flash Attention
saves memory, which allows larger batches, which improves GPU utilization,
which makes torch.compile's kernel fusion more effective. The whole is greater
than the sum of its parts.

### Where the time actually goes

For a single training step (125M model, batch=8, seq=1024, bf16):

```
Forward pass:        ~45%   (attention: ~25%, FFN matmuls: ~15%, other: ~5%)
Backward pass:       ~45%   (same operations, computing gradients)
Optimizer step:       ~5%   (AdamW parameter updates)
Data loading:         ~3%   (tokenized data from disk → GPU)
Python overhead:      ~2%   (with torch.compile; ~15% without)
```

Attention and FFN matrix multiplications dominate. This is why TF32, mixed
precision, and Flash Attention have the biggest impact — they directly
accelerate the operations that consume 80%+ of the step time.

## 10. Debugging: How to Disable Each Optimization

When training produces NaN losses, wrong results, or crashes, the first step is
isolating which optimization is causing the problem. Here is how to disable
each one independently:

### torch.compile

```bash
# Environment variable — no code change needed
COLA_NO_COMPILE=1 python scripts/train.py --config configs/tiny.yaml
```

The trainer checks this env var:

```python
# trainer.py:
if os.environ.get("COLA_NO_COMPILE") != "1":
    self.model = torch.compile(self.model, mode="default")
```

**When to disable:** Compilation errors, OOM during compilation, debugging
model logic (stack traces from compiled code are unreadable).

### TF32

```python
# Add to the top of your training script, or modify _setup_cuda_optimizations():
torch.backends.cuda.matmul.allow_tf32 = False
torch.backends.cudnn.allow_tf32 = False
```

**When to disable:** Suspected precision issues (loss not converging, gradients
going to zero). This is extremely rare for transformer training.

### Mixed precision

Set `precision: "fp32"` in your config YAML. The trainer will set both
`use_bf16` and `use_fp16` to False, and the `autocast` context manager becomes
a no-op.

```yaml
# configs/tiny.yaml:
training:
  precision: "fp32"  # disable mixed precision entirely
```

**When to disable:** NaN losses, suspected overflow/underflow, comparing against
a reference fp32 implementation.

### Flash Attention / SDPA backends

```python
# Force the slow math fallback (no Flash, no memory-efficient):
torch.backends.cuda.enable_flash_sdp(False)
torch.backends.cuda.enable_mem_efficient_sdp(False)
# This leaves only the math fallback: standard O(n²) attention
```

Or use the context manager for surgical testing:

```python
with torch.backends.cuda.sdp_kernel(
    enable_flash=False,
    enable_mem_efficient=False,
    enable_math=True,  # slow fallback
):
    output = model(input_ids)
```

**When to disable:** Suspected attention bugs, numerical differences vs
reference implementation, debugging attention patterns.

### Fused AdamW

```python
# In optimizer.py, change:
fused=torch.cuda.is_available()
# to:
fused=False
```

**When to disable:** Suspected optimizer bugs (extremely rare). The fused and
non-fused implementations should produce identical results.

### cuDNN benchmark

```python
torch.backends.cudnn.benchmark = False
```

**When to disable:** Variable-length inputs causing repeated benchmarking, or
suspected non-determinism (benchmark mode can select different algorithms
across runs, leading to tiny numerical differences).

### Non-blocking transfers

```python
# Change:
input_ids = batch["input_ids"].to(self.device, non_blocking=True)
# to:
input_ids = batch["input_ids"].to(self.device, non_blocking=False)
```

**When to disable:** Data corruption (tensor is used before transfer completes).
This should never happen with PyTorch's CUDA stream synchronization, but if you
see garbage values in your input tensors, this is the first thing to check.

### The "disable everything" debug configuration

If you are chasing a nasty bug, start with everything disabled and add
optimizations back one at a time:

```bash
COLA_NO_COMPILE=1 python scripts/train.py --config configs/tiny.yaml
```

With `precision: "fp32"` in the config and these lines in `_setup_cuda_optimizations()`:

```python
torch.backends.cuda.matmul.allow_tf32 = False
torch.backends.cudnn.allow_tf32 = False
torch.backends.cudnn.benchmark = False
torch.backends.cuda.enable_flash_sdp(False)
torch.backends.cuda.enable_mem_efficient_sdp(False)
```

This gives you pure fp32, no compilation, standard attention — the slowest but
most debuggable configuration. If the bug disappears, add optimizations back
one by one until it reappears. That is your culprit.

TS analogy: This is like adding `"use strict"` to every file, disabling all
babel transforms, and running in Node.js with `--no-opt` to find a subtle
runtime bug. Strip away every layer of optimization until the bug is
reproducible in the simplest environment, then add layers back.
