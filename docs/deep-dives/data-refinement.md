# Data Refinement: Making Your Training Data Better

The single most impactful thing you can do — beyond architecture, beyond
hyperparameters — is **improve your training data**. A 125M model trained on
clean, high-quality code will outperform a 350M model trained on garbage.

This guide covers what we've implemented, what else is possible, and what
the frontier looks like.

## 1. Why Data Quality Matters

There's a well-established result in ML research: **data quality scales better
than model size**. Concretely:

- The original GPT-3 (175B params) trained on unfiltered web data performed
  worse on many tasks than smaller models trained on curated data.
- Phi-1 (1.3B params) outperformed models 10x its size on code benchmarks,
  primarily because Microsoft curated high-quality "textbook-style" training data.
- LLaMA 1 showed that a 13B model trained on 1T high-quality tokens matched
  GPT-3 175B on most benchmarks.

The lesson: if you have limited compute (which we do on consumer GPUs),
maximizing data quality is the highest-leverage investment.

### The garbage-in-garbage-out problem

When your model trains on a file, it adjusts its weights to predict that
file's tokens more accurately. If the file contains:

```python
# bad code in training data
def f(x):
    try:
        pass
    except:
        pass
    a = 1
    b = 2
    c = 3
    d = 4
    e = 5
    # ... 200 more unused variables
```

The model spends training capacity learning to predict unused variable
declarations and bare except clauses. Those weights could have been used
learning actual useful patterns instead. Every bad file is a wasted
training step.

## 2. What's Already in StarCoderData

The BigCode team already applied significant filtering to create
StarCoderData from The Stack. Their pipeline includes:

- **Deduplication:** Removed near-duplicate files using MinHash/LSH.
- **License filtering:** Only code with permissive licenses (MIT, Apache, BSD, etc.).
- **PII removal:** Scrubbed emails, names, API keys, passwords.
- **Opt-out:** Removed code from developers who opted out of AI training.
- **Basic quality:** Removed very short files, empty files, binary files.

So the data is already pre-filtered. Our quality filter adds a second pass
that catches things the BigCode pipeline didn't focus on: minified code,
auto-generated files, syntax errors, data dumps, etc.

## 3. Our Quality Filter (Built-In)

The filter lives at `src/cola_coder/data/quality_filter.py` and is enabled
by default in `prepare_data.py`. It runs 10 checks on every file:

| Check | What It Catches | Default Threshold |
|-------|----------------|-------------------|
| **Length** | Too short (<5 lines) or too long (>10K lines) | 5 / 10,000 lines |
| **Avg line length** | Minified/bundled code | >200 chars avg |
| **Max line length** | Single huge lines (base64, data URIs) | >1000 chars |
| **Character diversity** | Repeated/degenerate content | <5% unique chars |
| **Auto-generated** | Files with "do not edit" / "generated by" headers | Keyword match |
| **Data files** | JSON dumps, CSV data, hex dumps | Pattern detection |
| **Comment ratio** | Files that are >85% comments | 85% threshold |
| **Test dumps** | Files that are >90% assert statements | 90% threshold |
| **Python parsing** | Python files with syntax errors | `ast.parse()` |
| **JS/TS balance** | JS/TS files with badly unbalanced braces | >20% imbalance |

### Design decisions

**Conservative by default.** If a check can't determine quality (unsupported
language, ambiguous content), it passes the file through. The filter only
rejects files it's confident are bad.

**Fail-safe.** If any check crashes on a file (unexpected encoding, etc.),
the file passes through. We never reject data we can't analyze.

**Fast.** All checks are string operations — no external tools, no network
calls, no heavy parsing libraries. The filter adds negligible overhead to
the data pipeline.

### Usage

```bash
# Default: filter enabled
python scripts/prepare_data.py --tokenizer tokenizer.json

# Disable filtering (use raw data)
python scripts/prepare_data.py --tokenizer tokenizer.json --no-filter
```

The filter prints a summary at the end showing how many files were kept,
rejected, and the breakdown by reason. Typical rejection rate on StarCoderData
is 5-15% depending on the languages selected.

## 4. Technique: Syntax Parsing

**What it is:** Run each file through the language's parser. If it doesn't
parse, the code has syntax errors — don't train on it.

**What we do:** Our filter uses Python's `ast.parse()` for Python files. For
JS/TS, we do a lightweight brace-balance check (a full parser would be a
heavy dependency).

**Going further:** If you wanted maximum coverage:

```python
# You could install language-specific parsers:
#   pip install tree-sitter tree-sitter-python tree-sitter-javascript
#
# tree-sitter is a universal parser framework. It can parse 40+ languages
# and tell you exactly if a file is syntactically valid.
#
# We didn't include it by default because it adds a native dependency,
# but it's the gold standard for multi-language syntax checking.
```

**Impact:** Medium. StarCoderData is already mostly parseable code. You'll
reject maybe 1-3% of files. But those files would have taught the model
broken syntax, so it's worth it.

## 5. Technique: Deduplication

**What it is:** Remove files that are identical or near-identical. Duplicated
code overweights those patterns during training — the model sees the same
code multiple times and memorizes it instead of generalizing.

**What StarCoderData already does:** MinHash/LSH near-deduplication. This
catches files that are 80%+ similar.

**Going further — exact deduplication:**

```python
# Hash each file and remove exact matches
import hashlib

seen_hashes = set()

def deduplicate(content: str) -> bool:
    """Return True if this file is unique (not seen before)."""
    h = hashlib.md5(content.encode()).hexdigest()
    if h in seen_hashes:
        return False
    seen_hashes.add(h)
    return True
```

This catches files that slipped through near-dedup (e.g., identical copies
across forks). Simple, fast, no false positives.

**Going further — content-level deduplication:**

Instead of file-level, deduplicate at the function/class level. If the same
function body appears in 100 different files, keep one copy. This is harder
to implement but more effective. Tools like SemGrep or tree-sitter can
extract function bodies for hashing.

**Impact:** Low-medium on StarCoderData (already deduplicated). High on
raw data sources.

## 6. Technique: Quality Scoring with Heuristics

**What it is:** Instead of binary keep/reject, assign a quality score to
each file and train preferentially on higher-scored files.

**Heuristic signals:**

| Signal | High Quality | Low Quality |
|--------|-------------|-------------|
| **Docstrings/comments** | Has docstrings on public functions | No documentation at all |
| **Variable names** | Descriptive (`user_count`, `maxRetries`) | Single letters (`a`, `x`, `tmp`) |
| **Function length** | 5-50 lines (focused, readable) | 500+ lines (God function) |
| **Import count** | Moderate (5-20 imports) | Huge (100+ imports = kitchen sink) |
| **Star count** | From popular repos (>100 stars) | From zero-star repos |
| **File path** | `src/`, `lib/`, `core/` | `vendor/`, `node_modules/`, `dist/` |

**Implementation approach:**

```python
def quality_score(content: str) -> float:
    """Score a file from 0.0 (garbage) to 1.0 (excellent).

    This is a rough heuristic, not a precise metric.
    """
    score = 0.5  # Start neutral

    lines = content.split("\n")
    non_empty = [l for l in lines if l.strip()]

    # Bonus: has docstrings
    if '"""' in content or "'''" in content or "/**" in content:
        score += 0.1

    # Bonus: reasonable function length (most functions < 30 lines)
    # Penalty: very long functions
    def_count = sum(1 for l in lines if "def " in l or "function " in l)
    if def_count > 0:
        avg_func_len = len(non_empty) / def_count
        if 5 < avg_func_len < 40:
            score += 0.1
        elif avg_func_len > 100:
            score -= 0.1

    # Bonus: descriptive variable names (average name length > 4)
    # (simplified — a real implementation would use AST)
    identifiers = re.findall(r'\b([a-zA-Z_]\w{0,30})\b', content[:5000])
    if identifiers:
        avg_name_len = sum(len(i) for i in identifiers) / len(identifiers)
        if avg_name_len > 5:
            score += 0.1

    # Penalty: too many empty lines (sloppy formatting)
    empty_ratio = (len(lines) - len(non_empty)) / max(len(lines), 1)
    if empty_ratio > 0.4:
        score -= 0.1

    return max(0.0, min(1.0, score))
```

**How to use scores:** Two options:

1. **Hard threshold:** Reject files below 0.3 (our filter already does
   something similar).
2. **Weighted sampling:** During training, sample files with probability
   proportional to their score. High-quality files get seen more often.

**Impact:** Medium-high. This is what Phi-1's "textbook quality" filtering
was doing (but with a much more sophisticated classifier).

## 7. Technique: Linting as a Quality Signal

**What it is:** Run a linter on the code and use the warning/error count
as a quality signal. Code with zero lint warnings is generally cleaner than
code with 50 warnings.

**Important:** You do NOT fix the code. You just use the lint score to
decide whether to keep it. Fixing code would change the distribution and
could introduce subtle bugs.

**For Python (ruff):**

```bash
# Score a file without modifying it
ruff check --statistics --output-format json my_file.py
```

Ruff is extremely fast (written in Rust) and can score thousands of files
per second. Count the number of warnings per line of code.

**Thresholds:**

```
0 warnings per 100 lines:    Excellent — keep, maybe upsample
1-5 warnings per 100 lines:  Fine — keep
5-20 warnings per 100 lines: Acceptable — keep but don't upsample
20+ warnings per 100 lines:  Noisy — consider rejecting
```

**Why we didn't include this by default:** It requires `ruff` to be
installed and would slow down the pipeline (subprocess call per file). For a
project where you need things to "just work or fall back," adding external
tool dependencies is a risk. But if you're doing a serious training run,
it's worth adding.

**For JavaScript/TypeScript:** ESLint can be used similarly, but it's much
slower than ruff and requires a Node.js environment.

**Impact:** Medium. Mostly helps filter out the worst offenders — files
with dozens of warnings tend to be prototype/scratch code.

## 8. Technique: LLM-as-a-Judge

**What it is:** Use an existing LLM (like Claude or GPT-4) to score code
quality. Send each file to an API and ask "Is this high-quality, well-written
code? Score 1-5."

**Why it's powerful:** LLMs can judge things that heuristics can't — code
clarity, good API design, proper error handling, idiomatic style.

**Why it's impractical at scale:** API costs. If you have 10 million files
and each costs $0.001 to evaluate, that's $10,000 just for scoring. And the
latency of 10M API calls would take days.

**Practical middle ground — score a sample:**

1. Sample 10,000 files randomly
2. Score them with an LLM ($10 cost)
3. Train a small classifier on the LLM scores
4. Use that classifier to score the remaining millions of files

This is what several research papers (Phi-1, StarCoder2) describe as
"LLM-annotated quality filtering." The small classifier is typically a
simple model (logistic regression on TF-IDF features) that approximates
the LLM's judgment at near-zero cost.

**Impact:** High. This is state-of-the-art for data filtering. But it
requires budget and engineering effort.

## 9. Technique: Curriculum Ordering

**What it is:** Instead of shuffling training data randomly, present it in
a deliberate order — typically from "easy" to "hard" or from "high quality"
to "lower quality."

**The idea:** Early in training, the model is learning basic patterns (syntax,
keywords, common structures). Showing it only clean, simple code during this
phase helps it build a solid foundation. Later, it can handle noisier data
and learn from more complex examples.

**Implementation:**

```python
# Sort training data by quality score before chunking
scored_files = [(quality_score(f), f) for f in all_files]
scored_files.sort(reverse=True)  # Best quality first

# First 30% of training: only top quality
# Middle 40%: all data
# Last 30%: emphasis on hard/complex examples
```

**Simpler version — quality epochs:**

```
Epoch 1: Train only on files scoring > 0.7 (top ~30%)
Epoch 2: Train on all files
Epoch 3: Train on all files, oversample complex code
```

**Impact:** Medium. Research results are mixed — curriculum helps for small
models but the effect diminishes at larger scales. Worth trying if you're
training the tiny or small config.

## 10. Technique: Data Mixing and Ratios

**What it is:** Control the proportion of each programming language in
your training data.

**Why it matters:** If your training data is 80% Python and 5% TypeScript,
the model will be much better at Python than TypeScript. The ratio directly
controls the model's strength across languages.

**Our default ratio:**

```yaml
# configs/small.yaml
data:
  languages: ["python", "typescript", "javascript", "java", "go", "rust"]
```

With no explicit weighting, you get whatever ratio StarCoderData provides
(roughly proportional to GitHub code volume).

**Custom mixing:**

```python
# Control the ratio explicitly
language_weights = {
    "python": 0.35,       # Largest share — most training data
    "typescript": 0.25,   # Second focus
    "javascript": 0.15,
    "go": 0.10,
    "rust": 0.10,
    "java": 0.05,
}
```

**Key insight from StarCoder research:** A 30/70 mix of "high quality / all
quality" works better than 100% high quality. The model needs volume to learn
rare patterns, even if some of that volume is noisier.

**Impact:** High for your specific use case. If you care most about Python
and TypeScript, overweight those. The model's performance tracks the training
distribution directly.

## 11. Technique: Synthetic Data Generation

**What it is:** Instead of (or in addition to) real code from GitHub,
generate training data using an existing LLM.

**Types of synthetic data:**

1. **Textbook-style code:** Ask an LLM to write clean, well-documented
   implementations of common algorithms and patterns. This is exactly what
   Microsoft's Phi-1 did, and it was remarkably effective.

2. **Problem-solution pairs:** Generate coding problems and their solutions.
   The model learns both "what constitutes a good problem" and "what a correct
   solution looks like."

3. **Refactored code:** Take real (messy) code, ask an LLM to clean it up,
   and train on the cleaned version. The model learns good style from the
   cleaned output.

4. **Diverse implementations:** For common tasks (sort a list, parse JSON,
   validate an email), generate 10 different implementations in different
   styles. This teaches the model multiple valid approaches.

**Practical approach:**

```python
# Using an LLM API to generate training data
prompt = """Write a clean, well-documented Python function that:
- Takes a list of integers
- Returns the k most frequent elements
- Handles edge cases properly
- Uses clear variable names
- Includes a docstring with examples
"""
# Generate 1000 such functions for different tasks
# Cost: ~$5 for 1000 generations with a fast model
```

**The catch:** Synthetic data can be repetitive and lack the diversity of
real code. The best results come from mixing synthetic data (20-30%) with
real data (70-80%).

**Impact:** High. Phi-1 proved that synthetic data can be the single most
effective lever for small models. But it requires API access and costs money.

## 12. Technique: Upsampling High-Quality Sources

**What it is:** Identify high-quality repositories or files and show them to
the model multiple times during training (upsampling), while showing
lower-quality data only once.

**How to identify high-quality repos:**

- **Star count:** Repos with >1000 stars are generally well-maintained.
- **Known good sources:** Standard libraries, popular frameworks, textbooks.
- **CI passing:** Repos with CI that consistently passes have working code.
- **Maintenance recency:** Recently updated repos are more likely to follow
  modern patterns.

**Implementation:**

```python
# Repeat high-quality files 3x in the training data
for file in training_files:
    if file.quality_score > 0.8:
        yield file  # Include 3 times
        yield file
        yield file
    else:
        yield file  # Include once
```

**Impact:** Medium-high. Simple to implement, measurable improvement. The
risk is overfitting to the upsampled files if you repeat too aggressively.
2-3x is safe; 10x is too much.

## 13. What NOT to Do

### Don't auto-format/fix code before training

It's tempting to run `black` or `prettier` on everything to make it
consistent. Don't.

- The model should learn to handle real-world formatting, not just one style.
- Auto-fixing can introduce bugs (rare but possible).
- It destroys useful signal: the model needs to learn that both `snake_case`
  and `camelCase` are valid in different contexts.

### Don't filter too aggressively

Removing the worst 10-15% of data helps. Removing the worst 50% hurts.
The model needs volume to learn rare patterns — obscure syntax, unusual
but valid constructs, domain-specific code.

**Rule of thumb:** If you're rejecting more than 20% of StarCoderData,
your filters are probably too aggressive.

### Don't over-represent one pattern

If you upsample "clean" code too much, the model becomes a textbook
generator — it writes beautiful, well-documented code that doesn't actually
work in production contexts. Real code is messy. The model needs to see some
messiness to be useful in real codebases.

### Don't train only on solutions

If you only show the model correct, complete code, it never learns what
partial code looks like. This hurts autocomplete quality. Include incomplete
files, files with TODOs, and work-in-progress code.

## 14. The Research Frontier

These are techniques used by leading labs but not yet practical for
individual developers:

### Data attribution and influence functions

Track which training examples actually improved the model's performance on a
benchmark, and which hurt it. Then remove the harmful examples and repeat.
This is computationally expensive (requires multiple training runs) but can
identify subtle data quality issues that heuristics miss.

### Contrastive filtering

Show the model pairs of code: one good, one bad. Train it to prefer the good
version. This is related to DPO (Direct Preference Optimization) and is used
in the RLHF pipeline for chat models. For code, the "good" version passes
tests and the "bad" version doesn't.

### Multi-epoch data scheduling

Instead of static data mixing, dynamically adjust the data mixture during
training based on the model's current capabilities. If the model already
handles Python well but struggles with Rust, automatically increase the Rust
proportion. This requires infrastructure to evaluate the model periodically
during training.

### Tool-use augmented data

Generate training data where the code interacts with tools (APIs, databases,
file systems) and include the tool outputs in the training sequence. This
teaches the model to use tools correctly — a capability that most code models
lack.

## 15. Practical Recommendations

Here's what to actually do, ordered by effort vs impact:

### Tier 1: Free, already done (just use the default pipeline)

- Our built-in quality filter catches the worst offenders.
- StarCoderData's existing deduplication and license filtering.
- Just run `make prepare` — you're already getting filtered data.

### Tier 2: Easy wins (a few hours of work)

- **Adjust language ratios** in your config to overweight the languages you
  care about most.
- **Add exact deduplication** (hash-based) — 20 lines of code.
- **Increase the tiny model's training data** before scaling up. Train tiny
  on the same data volume you'd use for small. It underfits (that's fine)
  but you validate the data pipeline.

### Tier 3: Moderate effort (a weekend project)

- **Add ruff-based quality scoring** for Python files. Filter out the worst
  20% by lint score.
- **Generate synthetic data** using Claude or GPT-4 for 1000-5000 high-quality
  code examples. Mix into training data at 10-20%.
- **Upsample known-good sources** (popular repos, standard library examples).

### Tier 4: Serious investment (days-weeks)

- **LLM-as-a-judge pipeline:** Score 10K files with an LLM, train a fast
  classifier, score the full dataset.
- **tree-sitter parsing** for multi-language syntax validation.
- **Curriculum ordering** experiments.
- **Contrastive filtering** using DPO on code pairs.

For this project, Tiers 1-2 are done. Tier 3 is the next step when you want
to squeeze more performance out of your model.
