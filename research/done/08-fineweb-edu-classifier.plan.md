# Research: FineWeb-Edu Style Code Quality Classifier

## Status: Practical — Proven approach, adapted for code

## How FineWeb-Edu Works

FineWeb-Edu (HuggingFace, 2024) solved a scaling problem: they wanted to filter 15T tokens
of web text for "educational quality" but couldn't afford to run an LLM on every document.

Their three-phase approach:

1. **Annotate**: Score ~450k web pages with LLaMA-3-70B-Instruct on a 0-5 quality scale
2. **Train classifier**: Fine-tune a small model (Snowflake-arctic-embed, 33M params) on those labels
3. **Score at scale**: Run the tiny classifier on all 15T tokens (1000x cheaper than LLM)

The result: filtering to score >= 3 produced FineWeb-Edu, which matched or beat much larger
curated datasets on educational benchmarks.

**Key insight**: You don't need the LLM to agree with itself perfectly. You need the
classifier to learn the *distribution* of what the LLM considers good. Even a noisy
classifier that's right 80% of the time massively improves data quality when applied
at scale, because the errors average out.

## Adapting for Code Quality

### Why this matters for cola-coder

Our existing quality_filter.py uses hand-crafted heuristics (line length, comment ratio,
syntax checking, etc.). These catch obviously bad code but miss subtler quality signals:

- Is this code well-structured and readable?
- Does it follow good practices for its language?
- Would a senior developer approve of this in a code review?
- Is it educational/instructive vs. just functional?

A learned classifier can capture these soft signals that are hard to write rules for.

### Best Pretrained Encoder for Code (2026)

| Model | Params | Pretraining | Code-specific | Speed |
|-------|--------|-------------|---------------|-------|
| CodeBERT (Microsoft) | 125M | Code + NL pairs | Yes | Medium |
| UniXcoder (Microsoft) | 125M | Code + AST + comments | Yes | Medium |
| StarEncoder (BigCode) | 125M | StarCoder data | Yes | Medium |
| CodeBERTa | 84M | Code only | Yes | Fast |
| DistilBERT | 66M | General text | No | Very fast |
| Snowflake-arctic-embed | 33M | General text | No | Fastest |

**Recommendation: Start with DistilBERT or CodeBERTa.**

Reasoning:
- FineWeb-Edu used a 33M general-text model and it worked great. The fine-tuning on
  labeled data matters more than the pretrained domain.
- DistilBERT (66M) is well-supported, fast, and we have plenty of fine-tuning data
  to teach it code-specific features.
- CodeBERTa (84M) is a reasonable upgrade if DistilBERT underperforms — it already
  knows code structure from pretraining.
- The 125M models (CodeBERT, UniXcoder) are overkill for a quality scorer. Save the
  VRAM for training the actual generative model.

### How Many Labeled Samples?

FineWeb-Edu used ~450k labeled samples, but they were classifying diverse web text.
Code quality is a more constrained domain:

- **5k-10k samples**: Minimum viable. Enough to learn basic quality patterns.
  Likely to underperform on edge cases.
- **10k-25k samples**: Sweet spot. Good generalization across languages and styles.
  This is what we should target.
- **25k-50k samples**: Diminishing returns for most use cases, but useful if
  scoring across many languages.
- **50k+**: Only needed if quality criteria are very nuanced or language-specific.

**Recommendation: 10k-20k samples across Python, TypeScript, and JavaScript.**

At $0.003/sample with Claude Haiku (or equivalent cheap model), that's $30-60 total.
Annotation can run in a few hours with batched API calls.

### Annotation Prompt Design

The prompt must produce consistent, well-calibrated scores. Key principles:

1. **Clear rubric**: Define each score level explicitly with examples
2. **Single dimension**: Score overall quality, not multiple axes
3. **Code-specific**: Mention code-specific quality signals
4. **Number-only response**: Reduces parsing complexity and cost

```
Rate this code on educational quality from 1 to 5:

1 = Garbage: broken syntax, autogenerated boilerplate, minified/obfuscated,
    data dumps, or meaningless snippets
2 = Poor: technically runs but terrible style — no structure, bad naming,
    no comments, copy-pasted spaghetti
3 = Average: functional code with some structure. Not great, not terrible.
    A typical file from a mid-level developer's side project.
4 = Good: clean, well-structured code. Meaningful names, some documentation,
    good use of language features. You'd accept this in a code review.
5 = Excellent: textbook-quality code. Clear structure, good documentation,
    idiomatic use of the language, educational value. You'd use this to
    teach someone the language.

Respond with ONLY the number (1-5).

```{language}
{code}
```
```

**Calibration tips**:
- Run the prompt on 100 diverse samples first, check distribution
- Ideal distribution: roughly 15% score 1, 25% score 2, 30% score 3, 20% score 4, 10% score 5
- If scores cluster too high, add "Be strict" to the prompt
- If scores cluster too low, add examples of what constitutes good code

### Trade-offs: Classifier vs. LLM-Judge vs. Heuristics

| Aspect | Heuristics | Classifier | LLM-Judge |
|--------|-----------|------------|-----------|
| Cost per file | ~0 | ~0.001ms | $0.001-0.01 |
| Speed (1M files) | Minutes | ~30 min | Days + $$$$ |
| Quality signals | Surface-level | Learned patterns | Deep understanding |
| Setup cost | Writing rules | $30-60 + training | Just a prompt |
| Maintenance | Manual rule updates | Retrain periodically | Prompt engineering |
| Accuracy on edge cases | Poor | Good | Best |
| Requires GPU | No | Yes (inference) | No (API) |
| Offline capable | Yes | Yes | No |

**Our strategy**: Use all three in a tiered approach:
1. **Heuristics first** (free): Catch obviously bad code — the existing quality_filter.py
2. **Classifier second** (cheap): Score remaining code on learned quality
3. **LLM-judge third** (expensive): Only for generating training data for the classifier

### Training Procedure

1. **Sample training data**: Pull 15-20k random code files from our existing dataset
2. **Annotate with LLM**: Score each with Claude Haiku using the rubric above
3. **Split**: 80% train, 10% val, 10% test
4. **Fine-tune**: Add a regression head (linear layer) on top of DistilBERT/CodeBERTa
5. **Train**: 3-5 epochs, lr=2e-5, batch_size=16, max_seq_len=512
6. **Evaluate**: Check correlation with LLM scores on held-out test set (target: r > 0.75)
7. **Deploy**: Export model, integrate as QualityClassifierFilter in the pipeline

Training takes ~30 minutes on a single GPU for 15k samples. The resulting model is
~250MB (DistilBERT) and can score ~1000 files/second on GPU, ~50 files/second on CPU.
