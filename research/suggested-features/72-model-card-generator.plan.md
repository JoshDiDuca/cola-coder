# 72 - Model Card Generator

## Overview

Automatically generate a `MODEL_CARD.md` from training manifests, evaluation history, and config files. Follows the HuggingFace model card format. Pulls training curves as ASCII charts. Auto-updates after each evaluation run. Output is a complete, human-readable document describing the model, its training, evaluation results, intended use, and limitations.

**Feature flag:** `config.model_card.enabled` (default: `true`; generates after each eval run and at training end)

---

## Motivation

A model card is the standard way to document an ML model's provenance, capabilities, and limitations. Without one:

- You forget what data the model was trained on after 3 months
- Sharing the model with someone else requires a verbal explanation
- Evaluating whether to use checkpoint X vs Y requires opening multiple files manually

With auto-generation from existing artifacts (manifest, eval history, config), writing a model card costs zero additional effort. It's generated by the tooling, not by you.

**HuggingFace format**: using the HF model card template makes Cola-Coder models immediately uploadable to the HuggingFace Hub if desired.

---

## Architecture / Design

### Data Sources

```
MODEL_CARD.md
├── model_description       ← config.yaml (model architecture)
├── training_data           ← manifest.yaml (dataset stats, version)
├── evaluation_results      ← eval_history.json (metrics at each step)
├── training_curves         ← eval_history.json → ASCII charts
├── intended_use            ← config.yaml + template
├── limitations             ← template + auto-detected from eval results
└── training_details        ← manifest.yaml (optimizer, LR, steps)
```

## Implementation Steps

### Step 1: Model Card Generator (`tools/model_card.py`)

```python
import json
import yaml
from pathlib import Path
from datetime import datetime
from typing import Optional

class ModelCardGenerator:
    def __init__(
        self,
        manifest_path: Path,
        eval_history_path: Path,
        config_path: Path,
        output_path: Path = Path("MODEL_CARD.md"),
    ):
        self.manifest_path = manifest_path
        self.eval_history_path = eval_history_path
        self.config_path = config_path
        self.output_path = output_path

    def _load_manifest(self) -> dict:
        if self.manifest_path.exists():
            return yaml.safe_load(self.manifest_path.read_text()) or {}
        return {}

    def _load_eval_history(self) -> dict:
        if self.eval_history_path.exists():
            return json.loads(self.eval_history_path.read_text())
        return {"history": [], "best": None}

    def _load_config(self) -> dict:
        if self.config_path.exists():
            return yaml.safe_load(self.config_path.read_text()) or {}
        return {}

    def _ascii_chart(self, values: list[float], title: str, width: int = 50) -> str:
        """Generate an ASCII line chart for a metric series."""
        if not values:
            return f"  {title}: (no data)\n"

        min_v = min(values)
        max_v = max(values)
        rng = max_v - min_v or 1

        blocks = "▁▂▃▄▅▆▇█"
        chart = "".join(
            blocks[min(7, int((v - min_v) / rng * 7))]
            for v in values
        )

        lines = [
            f"  **{title}**",
            f"  `{chart}`",
            f"  min={min_v:.4f}  max={max_v:.4f}  last={values[-1]:.4f}",
        ]
        return "\n".join(lines) + "\n"

    def _build_metrics_table(self, history: dict) -> str:
        """Build a markdown table of key metrics at best checkpoint."""
        best = history.get("best")
        if not best:
            return "_No evaluation data available._\n"

        rows = []
        # Find the eval entry at the best step
        for entry in history.get("history", []):
            if entry["step"] == best["step"]:
                for k, v in entry.get("metrics", {}).items():
                    rows.append((k, f"{v:.4f}"))
                break

        if not rows:
            return "_No metrics recorded._\n"

        lines = ["| Metric | Value |", "|--------|-------|"]
        for k, v in rows:
            lines.append(f"| `{k}` | {v} |")
        return "\n".join(lines) + "\n"

    def _detect_limitations(self, eval_history: dict, config: dict) -> list[str]:
        """Auto-detect limitations from eval results and config."""
        limitations = []

        history = eval_history.get("history", [])
        if history:
            last = history[-1]
            metrics = last.get("metrics", {})

            # Check syntax validity
            sv = metrics.get("syntax_validity_rate", 1.0)
            if sv < 0.8:
                limitations.append(
                    f"Syntax validity rate is {sv:.0%} — a non-trivial fraction of "
                    "generated outputs may not parse as valid TypeScript."
                )

            # Check type correctness
            tc = metrics.get("type_correctness_rate")
            if tc is not None and tc < 0.5:
                limitations.append(
                    f"Type correctness rate is {tc:.0%} — generated code frequently "
                    "contains type errors that would be caught by `tsc --strict`."
                )

        # Context window
        max_seq = config.get("model", {}).get("max_seq_len", 1024)
        limitations.append(
            f"Maximum context window is {max_seq} tokens. Code files longer than "
            "this are truncated."
        )

        # Training data
        limitations.append(
            "Trained exclusively on TypeScript/JavaScript. May generate plausible-looking "
            "but incorrect code for unfamiliar library APIs or framework-specific patterns."
        )

        limitations.append(
            "This is a research model trained from scratch. It is not intended for "
            "production use without careful human review."
        )

        return limitations

    def generate(self) -> str:
        manifest = self._load_manifest()
        eval_history = self._load_eval_history()
        config = self._load_config()

        model_cfg = config.get("model", {})
        training_cfg = config.get("training", {})
        data_cfg = manifest.get("data", {})

        model_name = config.get("model_name", "cola-coder")
        n_params = model_cfg.get("n_params", "unknown")
        n_layers = model_cfg.get("n_layers", "?")
        n_heads = model_cfg.get("n_heads", "?")
        d_model = model_cfg.get("d_model", "?")
        max_seq_len = model_cfg.get("max_seq_len", 1024)

        total_steps = manifest.get("training", {}).get("total_steps", "?")
        final_loss = manifest.get("training", {}).get("final_loss")
        dataset_version = manifest.get("data", {}).get("dataset_version", "unknown")
        total_tokens = data_cfg.get("total_tokens", "unknown")
        train_date = manifest.get("training", {}).get("started_at", "")[:10]

        # Training curves
        steps = [e["step"] for e in eval_history.get("history", [])]
        all_metrics = {}
        for entry in eval_history.get("history", []):
            for k, v in entry.get("metrics", {}).items():
                all_metrics.setdefault(k, []).append(v)

        curves_section = ""
        for metric, values in all_metrics.items():
            if len(values) >= 3:
                curves_section += self._ascii_chart(values, metric) + "\n"

        if not curves_section:
            curves_section = "_No training curve data available._\n"

        # Limitations
        limitations = self._detect_limitations(eval_history, config)
        limitations_md = "\n".join(f"- {l}" for l in limitations)

        # YAML frontmatter
        best = eval_history.get("best", {}) or {}
        best_metric_val = best.get("value", "")
        best_metric_name = best.get("metric", "")

        frontmatter = f"""---
language:
  - typescript
  - javascript
license: mit
tags:
  - code-generation
  - typescript
  - cola-coder
  - causal-lm
base_model: null
datasets:
  - custom-typescript-corpus
model_type: causal-lm
---
"""

        card = f"""{frontmatter}
# {model_name}

> Auto-generated model card. Last updated: {datetime.utcnow().strftime('%Y-%m-%d %H:%M UTC')}

## Model Description

Cola-Coder is a causal language model trained from scratch to generate TypeScript and JavaScript code.
It is a decoder-only transformer architecture, trained as part of the Cola-Coder research project.

**Architecture:**

| Parameter | Value |
|-----------|-------|
| Parameters | {n_params} |
| Layers | {n_layers} |
| Attention heads | {n_heads} |
| Model dimension | {d_model} |
| Max sequence length | {max_seq_len} tokens |
| Training steps | {total_steps} |
| Final training loss | {f"{final_loss:.4f}" if final_loss else "—"} |

## Training Data

- **Dataset version:** `{dataset_version}`
- **Training tokens:** {f"{total_tokens:,}" if isinstance(total_tokens, int) else total_tokens}
- **Sources:** TypeScript and JavaScript files from open-source repositories
- **Filters:** Minimum token length, deduplication via MinHash, quality scoring

For full dataset details, see `data/versions/{dataset_version}/metadata.yaml`.

## Evaluation Results

Best checkpoint: **step {best.get('step', '—')}** | {best_metric_name}: `{best_metric_val}`

{self._build_metrics_table(eval_history)}

## Training Curves

{curves_section}

## Training Details

- **Optimizer:** {training_cfg.get('optimizer', 'AdamW')}
- **Learning rate:** {training_cfg.get('lr', '—')}
- **Batch size:** {training_cfg.get('batch_size', '—')}
- **Gradient clipping:** {training_cfg.get('grad_clip', '—')}
- **Training date:** {train_date}
- **Hardware:** {manifest.get('hardware', {}).get('gpu', '—')}

## Intended Use

Cola-Coder is intended for:
- Research into code generation language models
- Benchmarking training techniques for TypeScript-specific models
- Completion assistance in development environments (via FastAPI server + VS Code extension)
- Educational exploration of from-scratch transformer training

It is **not** intended for:
- Security-critical code generation
- Production use without human review
- Non-TypeScript/JavaScript code generation

## Limitations

{limitations_md}

## Citation

If you use this model or codebase in your research, please cite:

```
@misc{{cola-coder,
  title = {{Cola-Coder: A From-Scratch TypeScript Code Generation Transformer}},
  year = {{2026}},
  url = {{https://github.com/your-repo/cola-coder}}
}}
```
"""
        return card

    def write(self) -> Path:
        card = self.generate()
        self.output_path.write_text(card, encoding="utf-8")
        return self.output_path
```

### Step 2: CLI Command

```python
# cola-coder model-card generate \
#   --manifest manifests/latest.yaml \
#   --eval-history eval_history.json \
#   --config config/training.yaml \
#   --output MODEL_CARD.md

def cmd_generate_model_card(args):
    gen = ModelCardGenerator(
        manifest_path=Path(args.manifest),
        eval_history_path=Path(args.eval_history),
        config_path=Path(args.config),
        output_path=Path(args.output),
    )
    path = gen.write()
    print(f"[model-card] Generated: {path}")
```

## Testing Strategy

1. **Generation test**: call `generate()` with known fixture YAML files, assert output contains model name, architecture params, and limitations section.
2. **ASCII chart test**: call `_ascii_chart` with monotonically decreasing loss values, assert output contains `▇` (high) on the left and `▁` (low) on the right.
3. **Missing data test**: call `generate()` with empty manifest and eval history, assert graceful output (no exceptions, placeholder text shown).
4. **Limitations detection test**: create eval history with syntax validity 0.5, assert generated limitations section mentions the low rate.
5. **Write test**: call `write()`, assert file exists and content matches `generate()` output.

## Performance Considerations

- File generation is synchronous and takes <100ms. No performance concerns.
- ASCII chart generation for 100 eval steps: O(N) string operations. Negligible.
- Large eval histories (1000+ entries with many metrics) may produce a large MODEL_CARD.md. Cap the number of history entries used in charts at 100 (every N-th entry).

## 2026 Best Practices

- **Auto-generate, not manual-write**: never write model cards by hand. Humans forget to update them when evaluation results change. Tie generation to the eval pipeline.
- **HuggingFace YAML frontmatter**: including the YAML metadata block makes the card immediately compatible with HF Hub upload tools (`huggingface_hub.ModelCard.push_to_hub()`).
- **Limitation detection from data**: auto-detect limitations from actual eval results (low syntax rate → note in limitations). This produces honest, accurate limitation statements rather than boilerplate.
- **Commit with model, not separately**: when saving a checkpoint, generate the model card at the same time and commit both together. Orphaned model cards (card without matching checkpoint or vice versa) are confusing.
- **ASCII charts over image links**: ASCII charts work in any markdown renderer, terminal, and GitHub. Base64 image embedding works in HuggingFace Hub but fails in many other contexts. Start with ASCII; optionally add images as a progressive enhancement.
